# MILPO — Multimodal Iterative Loop Prompt Optimization

> Version **4.4** — 2026-04-11

Le projet vise à créer une pipeline de classification multimodale servant à classifier les posts du média Views (@viewsfrance sur Instagram).

| 4.4 | 2026-04-11 | **Pivot data-centric + oracle**. Motivation : après une session d'audit empirique de la matrice de confusion du run 68 (baseline zero-shot, 70.9% vf), découverte que **54 annotations / ~2000 (2.7%)** étaient contradictoires avec la définition moderne de `post_mood` vs `post_news` — les variantes anciennes (< 2024) de post_news, où la news est portée uniquement par la caption sans texte overlay, étaient annotées comme `post_mood` par défaut visuel. Ce finding a deux conséquences. (1) **Les accuracies des runs 63-68 étaient sous-évaluées** d'environ 9pp : recalculées contre la vérité corrigée, le run 68 tombe à 61.6% (vs 70.9% rapporté). (2) **MILPO n'était pas fondamentalement bloqué par son algorithme, mais par le bruit d'annotation** — la boucle tentait d'optimiser un objectif calculé sur une ground truth contradictoire. **Observations méthodologiques documentées dans `docs/evaluation.md`** : (a) *biais de saillance textuelle* — quand une description de classe combine signal structurel obligatoire (ex. "flèche") et signaux sémantiques décoratifs (ex. "approfondi"), le classifieur match sur le sémantique ; (b) *variance inter-run du même ordre que les gains ciblés* à n=432 ; (c) *dérive taxonomique* entre deux époques éditoriales pré/post 2024 (187/437 test + 382/693 dev survivraient à un cut 2024). **Pipeline enrichi** : date de publication injectée dans le contexte classifieur via `PostInput.posted_at` propagée depuis `posts.timestamp` + `build_classifier_messages`, pour désambiguer les variantes temporelles de classes ; `reasoning_effort=high` sur le classifieur ; timeouts HTTP 20→60s et per-post 120→240s. **Outillage** : `scripts/analyze_run.sql` (10 blocs paramétrés par run_id/prev_run_id, split par époque, double score global vs production) + `scripts/validate_oracle.py` (Claude Sonnet 4.6 comme oracle indépendant sur les erreurs d'un run, output JSONL avec raisonnement complet, 4 buckets agrees_truth / agrees_prediction / proposes_other / parse_error). **Signal préliminaire oracle** sur 18/138 cas pré-2024 du run 70 : 50% agrees_truth (cascade corrigerait), 44% agrees_prediction (ground truth à re-auditer), 6% proposes_other. L'oracle se révèle **autant auditeur d'annotations que classifieur de secours** — il applique la taxonomie avec plus de rigueur que l'humain sur les cas limites. **Nouveau framing mémoire** : « Oracle-Based Annotations & Oracle-Based Multimodal Classification », data-centric AI appliqué à une taxonomie subjective à longue traîne. La boucle structurée v4.3 est conservée comme « piste explorée » pour la discussion. |
| 4.3 | 2026-04-11 | **Optimisation structurée à patches atomiques DSL**. Motivation : l'audit de la boucle ProTeGi a identifié 6 problèmes fondamentaux — espace de recherche infini (réécriture libre), pas de convergence, biais descripteur, évaluation bruitée, bandit décoratif, pas de holdout. Nouveau système : (1) **DSL formelle** avec 5 types de règles (`signal_to_label`, `disambiguation`, `priority`, `fallback`, `caption_policy`), vocabulaire fini de signaux (~30 FEED, ~17 REELS) extrait des descriptions taxonomiques, labels = enums taxonomie — seul le champ `criterion` ≤80 chars est du texte libre. (2) **Recherche locale structurée** : `optimize_slot` (critic→editor×3 patches→eval), `coordinate_ascent` (passes multiples vf→cat→strat). (3) **Objectif J unifié** = `0.6·macroF1_vf + 0.25·macroF1_cat + 0.15·acc_strat` évalué sur dev-opt fixe (70/30 stratifié, 612/249 posts). (4) **Propriétés défendables** : terminaison (patience+max_steps+max_passes), monotonie observée (cache `(post_id,axis,prompt_hash)` + temperature=0), optimum local empirique sur voisinage exploré, pas de cycles exacts (tabu par hash par slot, invalidé entre passes). (5) **Descripteur gelé** : optimisation uniquement des classifieurs aval, réouverture si >50% erreurs résiduelles = défaut de perception. **Migration 016** : `prompt_skeletons`, `prompt_rules`, `signal_vocabulary`, `dev_split_assignment`, `optimization_steps`. 15 fichiers créés, 4 modifiés, 69 nouveaux tests. Workflow : `scripts/run_structured_optimization.py`. |
| 4.2 | 2026-04-11 | **Mode SGD + fixes métriques descripteur**. Motivation : la boucle ProTeGi rejetait des candidats améliorants (+1.7%) à cause du seuil delta=2% et de la dilution de la métrique sur 3 axes pour les rewrites descripteur. (1) **Fix incumbent_records** : `async_multi_evaluate` comptait les matches de tous les bras (pas seulement l'incumbent) dans les métriques globales → accuracy >100%. Corrigé par guard `arm_id == incumbent_arm_id`. (2) **primary_axis** : pour les rewrites descripteur, la promotion est évaluée sur l'axe le plus erroné seul au lieu de moyenner les 3 axes. (3) **delta=0** par défaut : promotion si strictement mieux (`>`), comme en SGD classique. (4) **Mode `--sgd`** : évalue les candidats sur le batch d'erreurs lui-même au lieu d'une eval window séparée — pas de posts consommés, plus de steps possibles (B=10 par défaut). Analogie SGD complète : loss sur le mini-batch, pas de validation séparée. (5) **`--split test\|dev`** configurable. (6) **Loss `L(t)=1-accuracy`** affichée en TUI Rich + télémétrie WebSocket. |

| 3.4 | 2026-04-09 | **Implémentation de la boucle ProTeGi** (Pryzant et al. 2023, EMNLP, [arxiv 2305.03495](https://arxiv.org/abs/2305.03495)) dans `milpo/rewriter.py` et `scripts/run_simulation.py`. Trois primitives séparées comme dans le papier : (1) `compute_textual_gradient` (LLM_∇, *critic*) qui produit *m* critiques distinctes en JSON strict sans jamais éditer le prompt — le gradient textuel est matérialisé et persisté ; (2) `apply_gradient_edit` (LLM_δ, *editor*) qui prend le prompt, le gradient et les erreurs et génère *c* candidats édités, température 0.7 pour la diversité ; (3) `paraphrase_candidate` (LLM_mc, *paraphraser*) qui produit *p* paraphrases monte-carlo par candidat (skippé si *p* = 1, défaut pragmatique). Sélection finale par **Successive Rejects** (Audibert & Bubeck 2010, COLT), parameter-free, implémenté post-hoc dans `milpo/bandits.py`. Fonction `multi_evaluate` dans `scripts/run_simulation.py` qui évalue incumbent + N candidats en parallèle (ThreadPoolExecutor) sur les eval_window posts forward du cursor. **Migration 008** : tables `rewrite_gradients` (1 row par appel critic, contient `gradient_text`, `n_critiques`, modèle, tokens, latence) et `rewrite_beam_candidates` (1 row par candidat avec `parent_prompt_id`, `gradient_id` FK, `generation_kind` enum 'edit'/'paraphrase', `eval_accuracy`, `sr_phase`, `is_winner`). Hyperparams par défaut pragmatiques `m=3 c=4 p=1` (cost-driven, ~$21 par run dev complet) ; ablation paper-faithful `m=4 c=8 p=2` via `--protegi-paper-defaults`. Modèles `MODEL_CRITIC`/`MODEL_EDITOR`/`MODEL_PARAPHRASER` dans `milpo/config.py` (défauts vers `MODEL_REWRITER`, surchargeables via env vars `HILPO_MODEL_*`). Aucun impact sur la baseline B0 (run id=7) ni sur le DSPy baseline (`run_baseline.py` n'utilise pas le rewriter, `related_work/dspy_baseline/` reste isolé via `source='dspy_*'`).
